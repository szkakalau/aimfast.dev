# AimFast.Dev — 方法论文档

> 从信号到行动的操作手册 — 每条规则对应一个具体失败场景
>
> 版本：v1.0 | 基于 BuilderPulse E-P-A 框架

---

## 1. E-P-A 框架

```
信号吸入 → Evidence（证据锚定）→ Plain-English（白话翻译）→ Action（行动建议）
                                            ↓
                                      Counter-view（反方检验）
```

每个信号必须经历这三步，缺一不可：
- 没有证据的论断是噪音
- 没有翻译的信号无法行动
- 没有反方检验的判断是偏见

---

## 2. 证据锚定规范

### 铁律：数字 > 形容词

| ❌ 禁止 | ✅ 必须 |
|---------|---------|
| "很多人讨论X" | "X在HN有417条评论 + Lobsters有38条 + DEV有143条" |
| "Y最近很火" | "Y搜索量在Google Trends暴涨550%" |
| "Z值得关注" | "Z在GitHub 3天获得18,136 star，且出现在Reddit和HuggingFace" |

### 证据类型优先级（从高到低）

1. **讨论量**（评论数、帖子数）— 注意力的下限
2. **增长率**（搜索趋势、star 增速）— 方向的加速度
3. **跨平台确认**（同一话题出现在 ≥ 2 个独立平台）— 信号的真实性
4. **付费意愿信号**（Reddit 有人出价 $3、有人抱怨现有方案太贵）— 市场的真实存在

---

## 3. 白话翻译规范

翻译的目标不是"简化"，而是**换位**——站在不同读者的角度重新表达。

### 读者映射

| 读者类型 | 翻译方式 | 示例 |
|---------|---------|------|
| Builder（核心） | 这个信号意味着什么产品机会？ | "AI agent 数据泄露事件 → 企业需要一份 AI 文件访问审计报告" |
| Tech enthusiast | 这个技术为什么有意思？ | "新的路由协议让模型切换零成本" |
| Caution | 什么情况下这个信号是误导？ | "评论量可能被争议驱动，不代表付费需求" |

### 翻译铁律：把名词翻译成动词

不关心"AI agent"这个名词，关心的是这些动作：
`audit` `export` `verify` `recover` `route` `switch` `review` `monitor`

---

## 4. 反方观点制度化

每个推荐必须回答一个失败条件：

> **Counter-view**：[什么情况下这个判断是错的 / 什么情况下这个产品会失败]

这不是为了显得客观中立，而是一种**思维纪律**：
如果你说不清一个判断什么时候会失败，说明你还没理解它。

---

## 5. 机会筛选 — 三重过滤

### 第一重：证据密度
- 跨平台讨论量高（≥ 2 个独立源）
- 信号方向一致（不是同一个事件被同一批人转发）
- 搜索趋势明确上行

### 第二重：买方可识别性
- 能不能说出具体谁会付钱？"工程经理" > "开发者" > "所有人"
- 买方现在就在经历这个痛苦吗？有具体事件/事故吗？
- 买方有预算吗？还是只有抱怨没有购买力？

### 第三重：可快速验证
- 2 小时内能做出最小交付物（一个页面、一份报告、一个检查单）
- 不需要企业销售周期（不需要采购审批）
- 有自然分发路径（受影响的人已经在相关帖子里讨论了）

---

## 6. "谁先付钱"启发式

> 每个趋势，找出**谁因为不行动而承受最直接的代价**。那个人就是你的第一个付费用户。

- ❌ "所有人都在担心 AI 数据安全"
- ✅ "用了 Copilot 的工程经理，财务部看到账单比团队看到路由策略更早"

- ❌ "开发者想要更便宜的模型"
- ✅ "收到 $500+ 月账单的独立开发者，在 Reddit 上公开询问替代方案"

---

## 7. 不做什么

- 太泛的方向不做（"AI coding agent" 已经太宽泛）
- 需要法律合规的不做（跨司法管辖区）
- 需要实时体验才有说服力的不做（单人开发太重）
- 重复昨天新闻的不做（同一话题连续出现降权）

## 12. C 端消费机会识别（v2.1 新增）

> 背景：信号源（GitHub、HN、V2EX）天然偏开发者，评分公式的 actionability/buyer_clarity 维度历史上偏 B2B SaaS。这导致大量面向普通消费者的产品机会被系统性过滤。

### C 端机会的定义

C 端机会 = 买方是**普通用户**（非程序员、非工程经理），产品形态是**消费者工具/游戏/内容**（非 AI 开发套壳）。

### C 端信号特征识别清单

| 维度 | B 端信号（传统 AI SaaS） | C 端信号（消费者产品） |
|------|------------------------|---------------------|
| 买方 | 工程经理/CTO/开发者团队 | Mac 用户/上班族/学生/宠物主/旅行者/创意工作者 |
| 产品形态 | API/SaaS/CLI 工具/模板 | 桌面 App/Chrome 插件/游戏/订阅服务 |
| 定价 | $19-99/月/一次性 | $2.99-9.99 一次性 / $4.99-14.99/月 |
| 验证渠道 | Landing Page + HN/Reddit | App Store / Chrome Web Store / Reddit 消费者版块 / 社交媒体 |
| 关键词 | API, SaaS, CLI, agent, framework, template | game, mac app, chrome extension, desktop, audio, video, photo, health, home, pet, travel, book |
| 技术依赖 | 高（需要理解 MCP/RAG/API 等） | 低（用户不需要懂技术） |

### C 端信号的三类来源

1. **现有源中的 C 端信号**（process_signals.py 已自动标注 `c_end_flag=true`）
   - HN Show HN 中的消费产品（游戏、Mac App、桌面工具）
   - Product Hunt 中的非开发者工具
   - Reddit 中的消费话题
   - w2solo 中的副业/兼职故事

2. **从开发者信号中推导出的 C 端版本**
   - 例：一个开发者工具（Chrome DevTools MCP）→ C 端版本（自动网页截图 + PDF 导出，卖给普通用户）
   - 例：一个 AI 模型（Kokoro TTS）→ C 端版本（桌面有声书制作器）
   - 例：一个抱怨（Logitech Options+ 难用）→ C 端版本（外设配置管理 Mac App）

3. **待实现的独立 C 端采集源**（config.json `c_end_planned`）
   - Reddit 消费者子版块（BuyItForLife, InternetIsBeautiful, malelivingspace 等）
   - 小红书（中文 C 端趋势第一来源）
   - 豆瓣（书影音 + 生活方式）

### C 端验证方法（与 B 端不同！）

C 端产品**不要用 Landing Page** 验证。使用以下替代方法：
- **Mac App**: App Store 预注册页 + Reddit r/macapps 发帖
- **Chrome 插件**: Chrome Web Store 上架 + 相关论坛回答问题（签名档放链接）
- **游戏**: itch.io 发布 + HN Show HN（HN 偏好极简、有创意的个人游戏）
- **消费者服务**: Reddit 消费者子版块 DM / 小型 Kickstarter / Google Form 收集痛点
- **实物产品**: Etsy 预购 / 社交媒体展示制作过程

### 日报中的 C 端板块

从 v2.1 开始，每份日报必须包含独立的「🛍️ C端消费机会」板块，列出至少 3 个面向普通消费者的产品机会。该板块的优先级等同于传统的「发现机会」板块。

## 8. 信号打分公式

```
Score = cross_platform×3 + volume×2 + freshness×2 + actionability×2 + buyer_clarity×1
```

| 维度 | 1 分 | 2 分 | 3 分 | 5 分 |
|------|------|------|------|------|
| cross_platform | 1 平台 | — | 2 平台 | 3 平台+ |
| volume | < 50 | 50-200 | 200-500 | 500+ |
| freshness | 3 天+ | — | 昨天 | 今天 |
| actionability | 纯抱怨 | — | 方向模糊 | 具体产品+定价 |
| buyer_clarity | 不知道谁付 | — | 大概知道 | 具体角色 |

**触发 Action 方案的阈值：Score ≥ 15 且 cross_platform ≥ 3**

---

## 9. 信号衰减规则

- 连续 3 天出现的同一话题 → 降权 50%，不再作为头条
- 连续 5 天 → 自动归入"降温提醒"，不再作为信号
- 搜索引擎中 3 个月连续降温的关键词 → 仅作为背景参考

---

## 10. 验证管线

```
信号锁定 → Score≥15 → 2h Landing Page → 社区投放 → 7 天判断
                                            ↓
                              >100 UV + 注册 → 做产品
                              <30 UV → 放弃（写入经验库）
                              30-100 UV + 0 注册 → 调整或放弃
```

---

## 11. 日报五层结构

| 层级 | 名称 | 功能 |
|------|------|------|
| L1 | Today's One Build | 今日最强单一机会，可直接行动 |
| L2 | Top 3 信号 + Plain-English Brief | 快速扫描今天信号格局 |
| L3 | Discovery（发现层） | 新产品、搜索异动、涨星项目、抱怨热点 |
| L4 | Tech Radar + Competitive Intel + Trends | 技术雷达、竞争情报、趋势层 |
| L5 | Action（行动层） | 2h Build 方案 + 定价 + 验证步骤 + Counter-view |

## 12. 星球文章五类轮换

| 类型 | 占比 | 说明 |
|------|------|------|
| 机会深挖 | 40% | 今日最强信号 → 完整商业分析 |
| 反直觉洞察 | 25% | 今日最反直觉发现 → 为什么主流看法错了 |
| 方法论教学 | 20% | 拆解"我是怎么发现这个信号的" |
| 趋势预警 | 10% | 某个方向正在降温 → 提醒不要追高 |
| 失败复盘 | 5% | 之前推的机会验证失败了 → 为什么 + 学到了什么 |

**轮换规则**：连续 2 天同类型 → 第三天必须换。

# AimFast.Dev — 产品需求文档

> 从全球信号到高确定性行动 — 独立开发者的每日情报系统
>
> 版本：v1.0 | 日期：2026-05-27 | 基于 BuilderPulse 方法论重构

---

## 1. 产品概述

### 1.1 一句话定义

每日从全球 9+ 信号源中提取、翻译、验证高确定性机会，输出一份可直接行动的情报日报——不只告诉你"发生了什么"，而是告诉你"今天该做什么"。

### 1.2 产品定位：情报科，不是新闻聚合器

市面上不缺信息聚合工具。缺的是：

- **翻译层**：把技术信号翻译成商业机会
- **验证层**：跨平台三角验证，筛掉噪音
- **行动层**：每个机会附带 2 小时可构建的 MVP 方案 + 定价策略

**与 BuilderPulse 的关系**：本系统方法论文本上借鉴 BuilderPulse（刘小排）的 E-P-A 框架（Evidence → Plain-English → Action）和跨源三角验证方法论，但在以下方面做出差异化：

| 维度 | BuilderPulse | AimFast.Dev |
|------|-------------|---------------|
| 语言 | 中英双语独立输出 | 中文日报（核心）+ 关键机会英文摘要 |
| 信号源 | 9 个海外源 | 9 个海外源 + 5 个中国源 |
| 行动层 | 2h Build 建议 | 2h Build 建议 + Landing Page 方案 + 7 天验证追踪 |
| 定价 | 无 | 每个机会附定价策略 |
| 分发 | GitHub 开源 | 本地日报 + 智识星球文章 + [预留] 公众号/即刻分发 |

### 1.3 核心价值观

1. **狙击，不是扫射**：每天锁定 1 个最强机会，不是 100 个方向
2. **证据驱动，不是感觉驱动**：每条判断必须附带讨论量、搜索增长率、跨平台交叉确认
3. **翻译优先，不是术语自嗨**：每个技术信号必须翻译成"这对谁意味着什么"
4. **反方制度化，不是确认偏误**：每个推荐必须附带 Counter-view——什么情况下这个判断是错的
5. **行动闭环，不是信息消费**：日报的终点不是"知道了"，而是"今天可以做什么"

### 1.4 使用场景

- **每日 09:30**：情报科日报推送到主人面前，5 分钟读完
- **每日 10:00**：智识星球文章生成，主人审校后发布
- **信号≥3 交叉验证**：附带 Landing Page 方案 + 2h Build 规格
- **有流量/注册 → 做产品；没流量 → 放弃**：7 天验证周期，成本 1 小时

## 2. 情报方法论文档

### 2.1 E-P-A 框架（借鉴 BuilderPulse）

```
信号吸入 → Evidence（证据锚定）→ Plain-English（白话翻译）→ Action（行动建议）
                                              ↓
                                        Counter-view（反方检验）
```

**每个信号必须经历这三步**，缺一不可。没有证据的论断是噪音，没有翻译的信号无法行动，没有反方检验的判断是偏见。

### 2.2 证据锚定规范

```
不是"很多人讨论X" → 而是"X在HN有417条评论 + Lobsters有38条 + DEV有143条"
不是"Y最近很火"     → 而是"Y搜索量在Google Trends暴涨550%"
不是"Z值得关注"     → 而是"Z在GitHub 3天获得18,136 star，且出现在Reddit和HuggingFace"
```

证据类型优先级：
1. **讨论量**（评论数、帖子数）— 注意力的下限
2. **增长率**（搜索趋势、star 增速）— 方向的加速度
3. **跨平台确认**（同一话题出现在 ≥ 2 个独立平台）— 信号的真实性
4. **付费意愿信号**（Reddit 有人出价 $3、有人抱怨现有方案太贵）— 市场的真实存在

### 2.3 白话翻译规范

翻译的目标不是"简化"，而是**换位**——站在不同读者的角度重新表达：

| 读者类型 | 翻译方式 | 示例 |
|---------|---------|------|
| Builder（核心读者） | 这个信号意味着什么产品机会？ | "AI agent 数据泄露事件 → 企业需要一份 AI 文件访问审计报告" |
| Tech enthusiast | 这个技术为什么有意思？ | "新的路由协议让模型切换零成本" |
| Caution | 什么情况下这个信号是误导？ | "评论量可能被争议驱动，不代表付费需求" |

**翻译铁律**：把名词翻译成动词。不关心"AI agent"这个名词，关心的是"audit（审计）、export（导出）、verify（验证）、recover（恢复）"这些动作。

### 2.4 反方观点制度化

每个推荐必须回答一个失败条件：

> **Counter-view**：[什么情况下这个判断是错的 / 什么情况下这个产品会失败]

这不是为了显得客观中立，而是一种**思维纪律**——如果你说不清一个判断什么时候会失败，说明你还没理解它。

### 2.5 机会筛选的三重过滤

借鉴 BuilderPulse 的筛选逻辑：

**第一重：证据密度**
- 跨平台讨论量高（≥ 2 个独立源）
- 信号方向一致（不是同一个事件被同一批人转发）
- 搜索趋势明确上行

**第二重：买方可识别性**
- 能不能说出具体谁会付钱？"工程经理" > "开发者" > "所有人"
- 买方现在就在经历这个痛苦吗？有具体事件/事故吗？
- 买方有预算吗？还是只有抱怨没有购买力？

**第三重：可快速验证**
- 2 小时内能做出最小交付物（一个页面、一份报告、一个检查单）
- 不需要企业销售周期（不需要采购审批）
- 有自然分发路径（受影响的人已经在相关帖子里讨论了）

**不做什么同样重要**：
- 太泛的方向不做（"AI coding agent" 已经太宽泛）
- 需要法律合规的不做（"预测市场监控器" 跨司法管辖区）
- 需要实时体验才有说服力的不做（"协作式音频编辑器" 单人开发太重）
- 重复昨天新闻的不做（同一话题连续出现降权）

### 2.6 "谁先付钱"启发式

这是 BuilderPulse 最核心的分析工具：

> 每个趋势，找出**谁因为不行动而承受最直接的代价**。那个人就是你的第一个付费用户。

- 不是"所有人都在担心 AI 数据安全" → 而是"用了 Copilot 的工程经理，财务部看到账单比团队看到路由策略更早"
- 不是"开发者想要更便宜的模型" → 而是"收到 $500+ 月账单的独立开发者，在 Reddit 上公开询问替代方案"

### 3.3 Landing Page 方案（信号≥3 交叉验证时附带）

保留原有 Landing Page 方案模板（来自 `机会狙击系统.md`），但增加以下字段：

- **信号→产品映射**：为什么这个信号对应这个产品形态
- **竞争定位**：与现有竞品的差异化（不是功能对比，是"这个产品在谁的 workflow 中替代了什么"）
- **定价锚点**：来自 Competitive Intel 中同类产品的真实定价参考
- **反方检验**：这个 Landing Page 方案什么情况下无效

**语言铁律（2026-05-27 加入）**：

> Landing Page 的语言必须与**信号来源媒体的语言**一致，禁止混用。

- 信号来自英语媒体（HN、Reddit、Product Hunt、GitHub、Indie Hackers 等）→ Landing Page **全部英语**
- 信号来自中文媒体（V2EX、即刻、小红书、知乎等）→ Landing Page **全部中文**
- 信号同时来自中英文媒体（跨平台交叉验证）→ 以**目标付费用户使用的语言**为准

原因：LP 的第一批访客来自信号来源帖子的读者——他们用什么语言读帖子，就用什么语言看 LP。中英混用破坏信任感，降低转化率。

### 3.4 智识星球文章（每日公开发布）

日报是内部决策工具，星球文章是**对外建立权威、吸引付费用户的内容产品**。两者源自同一套信号，但处理方式完全不同。

| 维度 | 情报日报 | 智识星球文章 |
|------|---------|-------------|
| 读者 | 主人（1 人） | 星球成员 + 潜在付费用户 |
| 目的 | 快速决策"今天做什么" | 建立权威 → 吸引付费 → 留存成员 |
| 长度 | 1500-3000 字 | 2000-4000 字 |
| 结构 | 五层数据漏斗 | 叙事驱动，单话题深挖 |
| 语气 | 冷静、证据优先 | 有观点、有故事、有温度 |
| 发布 | 本地归档 | 智识星球 APP / Web |
| 频率 | 每日 | 每日（尽量不断更） |

#### 3.4.1 文章类型

每天从当日信号中选择一个角度切入，类型轮换保持新鲜感：

| 类型 | 占比 | 说明 | 示例标题 |
|------|------|------|---------|
| **机会深挖** | 40% | 今日最强信号 → 完整商业分析 | "一个 417 条评论的 HN 帖子，藏着一个 $19/次的 SaaS 机会" |
| **反直觉洞察** | 25% | 今日最反直觉发现 → 为什么主流看法错了 | "AI 让开发变快了，但最值钱的机会在'变慢'那边" |
| **方法论教学** | 20% | 拆解"我是怎么发现这个信号的" | "我是怎么从 3 个不相关的帖子中拼出一个 $5K MRR 机会的" |
| **趋势预警** | 10% | 某个方向正在降温 → 提醒不要追高 | "上个月所有人都在追的 X，搜索量已经跌了 60%" |
| **失败复盘** | 5% | 之前推的机会验证失败了 → 为什么 + 学到了什么 | "这个 Landing Page 有 300 UV 但 0 注册，我错在哪" |

**轮换规则**：连续 2 天同类型 → 第三天必须换；机会深挖和反直觉洞察之间交替为主。

#### 3.4.3 写作规范（星球文章专用）

**标题铁律**：
- 必须有数字或具体名词（"417 条评论" > "一篇热帖"）
- 必须制造好奇心缺口（读完标题想知道答案）
- 禁止标题党——标题承诺的内容正文必须兑现
- 长度 ≤ 30 字

**开头铁律**：
- 前 3 句决定读者是否继续读
- 禁止"最近，随着 AI 技术的发展……"式开头
- 用场景/故事/人物切入，不用概念切入
- 第一段 ≤ 4 句话

**正文铁律**：
- 每个观点必须有证据（数字、链接、截图、引用）
- 禁止"很多人认为"——说出具体是谁
- 禁止"这可能是一个机会"——说出具体是什么机会、谁付钱、多少钱
- 每 300 字至少一个具体数字或引用
- 段落短（≤ 4 句），多用分行和加粗制造阅读节奏

**结尾铁律**：
- 必须有"如果是我"的具体行动——不说"值得关注"，说"我会这样做"
- 星球文章不做硬广，用"这篇分析的完整数据和 Landing Page 模板在星球里"做自然引流
- 下一篇预告必须具体（不能写"敬请期待"）

**语气铁律**：
- 像一个做了 10 年产品的朋友在跟你聊天，不像一个分析师在出报告
- 不装、不吹、不贩卖焦虑
- 允许说"我不确定"、"我上次猜错了"、"这个方向我也没看懂"
- 核心读者画像是：独立开发者、想做副业的程序员、MicroSaaS 创业者

#### 3.4.4 星球运营策略

**内容节奏**：
- 每日免费文章（公开，全网分发）→ 建立 top-of-mind
- 星球内额外内容：完整日报 + Landing Page 模板 + 机会追踪表 + 失败案例库

**转化漏斗**：
```
免费文章（每日） → 读者觉得有用 → 关注/收藏
                              ↓
                      想看完整日报 + 模板 → 加入星球
                              ↓
                      星球内持续获得价值 → 续费 + 口碑推荐
```

**免费 vs 付费边界**：
| 内容 | 免费（公开文章） | 付费（星球内） |
|------|----------------|--------------|
| 今日信号分析 | 1 个深度分析 | 全部 Top 3 + Brief |
| 机会解读 | 有 | 更详细 + 竞品对比 |
| 行动方案 | "如果是我"方向性建议 | 完整 2h Build 规格 + 定价策略 |
| Landing Page 模板 | 无 | 有（可复制粘贴） |
| 机会追踪 | 无 | 完整追踪表 + 失败复盘 |
| 完整日报 | 无 | 每日完整五层日报 |

**不做什么**：
- ❌ 不刷屏——每天只发 1 篇
- ❌ 不贩卖焦虑——"再不行动就晚了"这类话术绝对禁止
- ❌ 不推垃圾工具——没有自己验证过的方向，不推荐具体产品
- ❌ 不建群——星球就是主阵地，不分流到微信群

## 4. 信号采集系统

### 4.1 海外信号源

| 信号源 | 采集内容 | 采集方式 | 优先级 |
|--------|---------|---------|--------|
| Hacker News | Top stories + 评论数 + Show HN | Algolia API | P0 |
| GitHub Trending | 热门 repo + star 增速 | 爬虫 / API | P0 |
| Google Trends | 搜索词暴涨/降温 + 相关查询 | pytrends | P0 |
| Product Hunt | 当日新品 + upvote + 评论 | API | P1 |
| HuggingFace | 热门模型 + 下载量 + 趋势 | API | P1 |
| Reddit | r/programming, r/MachineLearning, r/SaaS 热帖 | RSS / API | P1 |
| Indie Hackers | 产品发布 + 收入讨论 + MRR 相关 | RSS | P1 |
| Lobsters | 技术讨论热帖 | RSS | P2 |
| DEV Community | 热门文章 + 标签趋势 | API | P2 |

### 4.2 中国信号源

| 信号源 | 采集内容 | 采集方式 | 优先级 |
|--------|---------|---------|--------|
| V2EX | 创意/分享区热帖 | API | P0 |
| 即刻 | 创业/副业/独立开发圈热门 | mobile_use / API | P0 |
| 小红书 | 创业/副业爆款 + 搜索趋势 | 爆款检测 skill | P0 |
| TikTok 热销品 | 海外热销品 + GMV + 趋势 | 已有定时任务 | P1 |
| 知乎 | 创业/技术热问 | 搜索 | P2 |

### 4.3 采集规范

- 每条信号：来源 URL + 标题 + 讨论量 + 采集时间 + 正文摘要
- 同一事件跨平台 → 合并为一条，标注多平台验证
- 原始数据保留 7 天，日报和信号数据永久保留
- 采集时间：每日 08:00 Shanghai Time（海外信号源前一天的数据 + 中国信号源当日早间数据）

### 4.4 信号打分公式（v2.0 升级版）

```
Score = cross_platform×3 + volume×2 + freshness×2 + actionability×2 + buyer_clarity×1
```

| 维度 | 评分规则 |
|------|---------|
| cross_platform | 1 平台 = 1 分, 2 平台 = 3 分, 3 平台+ = 5 分 |
| volume | < 50 讨论 = 1 分, 50-200 = 2 分, 200-500 = 3 分, 500+ = 5 分 |
| freshness | 今天 = 5 分, 昨天 = 3 分, 持续 3 天以上 = 1 分（降权，不再新鲜） |
| actionability | 能推导出具体产品 + 定价 = 5 分, 方向模糊 = 3 分, 纯抱怨 = 1 分 |
| buyer_clarity | 能说出具体谁付钱 = 5 分, 大概知道谁会买 = 3 分, 不知道谁会付钱 = 1 分 |

**触发 Action 方案的阈值：Score ≥ 15 且 cross_platform ≥ 3**

### 4.5 信号衰减规则

- 连续 3 天出现的同一话题 → 降权 50%，不再作为头条
- 连续 5 天 → 自动归入"降温提醒"，不再作为信号
- 搜索引擎中 3 个月连续降温的关键词 → 仅作为背景参考

## 5. 机遇验证管线

### 5.1 狙击循环（从原有系统保留）

```
信号锁定 → 情报日报输出 → Score≥15 → 2h Landing Page → 扔到痛点现场 → 7 天见生死
   ↑                                                                        │
   │                                                              ┌─────────┴──────────┐
   │                                                              │                    │
   │                                                           有流量+注册           没流量
   │                                                              │                    │
   │                                                         做产品+建流量         果断放弃
   │                                                              │                    │
   │                                                        自然流量飞轮        1h 成本，写入经验库
   │                                                              │                    │
   └──────────────────────────────────────────────────────────────┘                    │
                                    ↑                                                   │
                                    └─── 经验反哺信号判断 ←──────────────────────────────┘
```

### 5.2 验证阶段（7 天）

**第一阶段：社区投放（Day 1-7，零成本验证）**

信号来源本身就是流量入口。不买广告，不做 SEO，直接去痛点现场摆解决方案：

- 海外机会 → HN/Reddit/DEV Community 相关帖子回复
- 国内机会 → V2EX/即刻/小红书 相关话题下发帖

| 步骤 | 操作 | 话术要点 |
|------|------|---------|
| 1 | 找到信号来源帖 | 先读完原帖和热门评论，确保理解痛点 |
| 2 | 写有价值的回复 | 先认同痛点，再说"我做了 XX，可以帮你 YY" |
| 3 | 附 Landing Page 链接 | 给上下文，不直接丢裸链接 |

**第二阶段：判断（Day 7）**

| 数据 | 判断 | 行动 |
|------|------|------|
| > 100 UV 且有注册/咨询 | 验证通过 | 投入做产品 + SEO + 内容 |
| < 30 UV | 验证失败 | 问题不痛或方案不对，放弃 |
| 30-100 UV 但零注册 | 流量对但转化差 | 调整 Landing Page 或放弃 |

**第三阶段：验证通过后的长期流量（Day 7+）**
- SEO 长尾词：针对具体痛点关键词做内容页
- 微内容引流：写教程/文章发布到 Medium/知乎/DEV
- Product Hunt 正式发布
- 付费广告（仅当 LTV 测算成立）

### 5.3 Landing Page 部署策略（保留原有方案）

- 域名：一个主域名 + 子目录（`aimfast.dev/project-name/`）
- 平台：Vercel（免费，自动部署）
- 放弃的页面不删，标注"实验已结束"，积累长尾流量反哺主域名

### 5.4 数据采集与分析（2026-05-27 更新）

#### 5.4.1 当前状态

已使用 **Vercel Web Analytics**（`@vercel/analytics` package）在 Landing Page 中嵌入分析代码，Vercel Dashboard 可查看页面访问量、访客数、来源等基本指标。

**已知限制**：Vercel Web Analytics **不提供 REST API**。截至 2026 年 5 月，无法通过 API / CLI / MCP 程序化读取分析数据。社区功能请求持续开放中，Vercel 产品团队已接收反馈但尚未发布。

这意味着 `update_tracking.py` 中的 `day7_uv` 和 `day7_signups` 目前**无法自动填充**，需要人工从 Vercel Dashboard 查看后手动更新到 `tracking/opportunities.json`。

#### 5.4.2 目标架构（规划中，暂不实施）

```
Vercel Web Analytics
       │
       ▼

       │
       ▼

       │
       ▼

       │
       ▼

```

#### 5.4.3 Drains 事件数据格式

每个 pageview 事件包含的字段（可用于分析）：

| 字段 | 用途 |
|------|------|

| `sessionId` / `deviceId` | 去重 UV / 回访识别 |
| `country` / `city` | 地理分布 |
| `referrer` | 流量来源（可追溯到信号来源帖） |
| `eventType` | `pageview` 或自定义 `event`（如注册点击） |
| `queryParams` | UTM 参数追踪 |

## 附录 A：BuilderPulse 方法论核心摘要

以下是 BuilderPulse（刘小排）45 天日更中提取的方法论精华，作为本系统的设计参考：

1. **E-P-A 铁律**：每个信号必须经历 Evidence（证据）→ Plain-English（白话翻译）→ Action（行动）三步，缺一不可
2. **Counter-view 制度化**：每个判断必须附带一个明确的失效条件
3. **谁先付钱启发式**：找出"因不行动而承受最直接代价的人"，而非"所有关心这件事的人"
4. **工作流中心主义**：不追模型或平台，只关注 Builder 能交付的"工件"（report、checklist、receipt、ledger）
5. **动词优先**：不关心名词（AI agent），只关心动词（audit、export、verify、recover、route、switch）
6. **音量 ≠ 价值**：讨论量是注意力的下限，但不是付费意愿的信号；小的付费信号（Reddit 上有人出价 $3）可能比大的讨论量更有信息量
7. **价格可见性**：一个机会从模糊讨论变成可行动，关键在于价格能否变成一行 spreadsheet
8. **衰减意识**：昨天的大新闻今天就应该降权；连续 3 天的同一话题是回声，不是趋势
9. **双声部写作**：每个段落同时服务两个读者——Enthusiast 看白话翻译，Builder 看行动建议
10. **翻译者角色**：作者不是分析师，是将技术信号翻译为商业机会的中间人

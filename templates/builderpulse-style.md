## 核心写作铁律

### 1. 人话开场 (刘小排说)
- 每天以一个**有人味的总编点评**开场（~200 字）
- 结构：大家都在聊X → 但真正可构建的信号是Y → 谁会先付钱 → 为什么是这周 → $19报告值不值 → 真正麻烦的活是什么
- **不写**"今日AI发展迅速"类空洞话

### 2. 四段式原子结构 (每个子节必须遵守)
```
🔍 信号 → 白话说 → 关键判断 → 反向视角
```
- **信号**: 具体数据+链接，绝不模棱两可
- **白话说**: 把技术信号翻译成Builder能行动的人话（核心卖点）
- **关键判断**: 一个明确的、可争论的建议
- **反向视角**: 什么情况下这个判断是错的 / 风险是什么

### 3. 证据铁律
- 每个判断必须附带**具体数字**（评论数、star数、金额、百分比）
- 禁词: "很多人""最近很火""大家觉得"
- 引用格式: "引发 XXX 条讨论" / "上涨 +X%"

### 4. 买家命名
- 每个推荐必须回答: **谁会最先付钱？**
- 具体到角色: "小型产品团队的工程负责人"而非"企业"
- 回答: **为什么是这周？** (时效性理由)

### 5. 2小时构建公式
- 产品名（英文 + 中文解释）
- 一句话描述
- 支撑证据（讨论量）
- 为什么不选另外两个（显式排除）
- 定价锚点: $19 首次 / $9-29/月
- 最快验证路径: 今天就能做的验证动作
- MVP保持人工: Google Form + Markdown即可

### 6. 白话简报格式
- 一句核心判断
- 证据表格: 证据 | 讨论量 | 白话含义
- 读者行动表: 技术爱好者 | Builder | 谨慎点

### 7. 定价意识
- 每个推荐必须有定价锚点
- "第一版保持人工"是核心原则
- 一次性报告 → 重复订阅的升级路径
- 单位定价: 按追回的钱、省下的时间、或避免的成本来定价

## 日报结构（BuilderPulse 标准）

1. **刘小排说** — 人话编辑开场
2. **今日 2 小时构建** — 产品名 + 一句话 + 证据
3. **今日 Top 3 信号** — 复合信号（每个信号连接多个数据点）
4. **白话简报** — 核心判断 + 证据表 + 读者行动表
5. **发现机会** (4个子节):
   - Solo-founder 产品发布
   - 搜索词暴涨
   - GitHub 快速增长开源项目
   - 开发者抱怨
6. **技术选型** (5个子节):
   - 大公司关停/降级
   - 增长最快开发者工具
   - HuggingFace 最热模型
   - 开源 AI 进展
   - Show HN 技术栈
7. **竞争情报** (3个子节):
   - Indie 开发者收入与定价
   - 沉寂项目复活
   - "XX已死"或迁移文章
8. **趋势判断** (4个子节):
   - 本周最常见关键词
   - VC/YC 关注话题
   - 降温 AI 搜索词
   - 新词雷达
9. **行动触发** (4个子节):
   - 2小时/周末做什么（详细版）
   - 定价与变现模型
   - 最反直觉发现
   - Product Hunt + 开发者工具重叠

## 星球文章风格

### 标题
- ≤ 30 字
- 具体数字或对比
- 制造好奇心缺口

### 正文
- 第一句 = 具体场景/人物/数据点
- 每 300 字至少一个具体数字或引用
- 观点 → 证据 → 白话翻译 → 行动启示
- 像有经验的 Builder 在聊天，不是分析师报告

### 结尾
- "如果是我"的具体行动
- 失败条件: 什么情况下这个判断是错的

### 语气
- 不装、不吹、不贩卖焦虑
- 承认不确定性

## 白话翻译技巧

| 技术术语 | 白话表达 |
|---------|---------|
| AI agent | 能代表用户调用工具和采取步骤的软件 |
| self-hosted | 软件跑在你自己的服务器上，而不是供应商那里 |
| local-first | 文件先在自己的机器上可用，云端分享是可选的 |
| RAG | 在生成AI答案前先搜索你自己的数据 |
| eBPF | 在Linux内核里运行的小型受验证程序 |
| MCP | AI工具访问应用和数据的连接器标准 |
| CSP | 浏览器用来规定页面可以加载哪些代码和资源的规则 |
| ONNX | 一种可移植模型格式 |

**原则**: 每次引入技术术语时，必须在同一句中用白话解释。不假设读者知道任何缩写。

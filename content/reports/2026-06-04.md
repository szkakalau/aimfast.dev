# 🎯 今日 2 小时构建

## SkillPack — AI 编码规范打包与审计工具

**一句话描述**：把你的 `.claude` 或 `.cursorrules` 文件打包成可复用的“技能包”，并生成一份报告告诉你团队的 AI 代码质量风险在哪里。

**支撑证据**：
- `mattpocock/skills` 在 GitHub 获得 **116,560 star**，讨论量巨大（但无商业版本）
- `shanraisshan/claude-code-best-practice` 获得 **28k star**，用户急求最佳实践
- HN 上关于 “Uber's $1,500/month AI limit” 的帖子有 **488 条评论**，大家都在讨论 AI 代码的 ROI 和风险

**为什么不选另外两个方向？**

1.  **Hermes Agent / 通用 Agent 框架**：太卷了。NousResearch 有 179k star，Datawhale 有 56k star。作为独立开发者，你无法在“通用框架”赛道竞争。你需要的是一个**垂直、有明确买家、2 小时能验证**的东西。
2.  **AI 记忆系统（MemPalace）**：赛道很热，但买方不清晰。谁为“AI 记忆”付费？是应用开发者（B2B）还是终端用户（B2C）？对于独立开发者，B2C 太难，B2B 需要和 LangChain 等生态竞争。证据密度不够。

**定价**：
- **$19**：一次性项目级 AI 编码规范审计报告（上传你的项目，分析 AI 生成代码的风险 + 给出定制 `.claude` 技能包）
- **$9/月**：监控订阅，当你的项目规则或依赖更新时，自动推送技能包更新和风险报告

**最快验证路径（今天就能做）**：
1.  **2 小时内**：用 Google Form 创建一个“AI 编码规范审计请求”表单。问题包括：项目语言、团队规模、是否使用 Cursor/Claude Code、最大的痛点是什么。
2.  **发布到**：`mattpocock/skills` 的 GitHub Issues（询问“有人需要打包好的技能包吗？”）、HN 上 Uber 帖子的评论区（作为“解决方案”回复）、以及相关 Reddit 子版块（r/cursor, r/ClaudeAI）。
3.  **目标**：24 小时内获得 10 个表单提交，其中 3 个愿意付费 $19 进行审计。

**MVP 保持人工**：你不需要写代码。收到表单后，手动分析项目结构，用你现有的 `.claude` 知识写出一个“技能包草案”，并用 Markdown 输出一份报告。如果 7 天后有 5 个付费用户，再开始构建自动化工具。

# 📊 今日 Top 3 信号

## 信号 1：AI 编码技能包（Skills / Best Practices）爆发

- **证据**：`mattpocock/skills`（116k star）、`shanraisshan/claude-code-best-practice`（28k star）、`addyosmani/agent-skills`（28k star）同时出现在 GitHub Trending。这不是巧合，是生态成熟信号。
- **白话含义**：开发者已经过了“AI 能不能写代码”的兴奋期，进入了“如何让 AI 写出符合我项目标准的好代码”的务实期。他们需要**可复用的规则和模板**。
- **关键判断**：这是一个**工具市场**的机会，不是框架市场。不要造另一个“AI agent”，而是造“AI agent 的配置管理器”。
- **反向视角**：如果大公司（如 GitHub、Cursor）直接内置了“技能包市场”功能，这个独立产品的窗口期可能只有 3-6 个月。风险在于**平台绑架**。

## 信号 2：Uber $1,500/月 AI 额度引发的定价与 ROI 焦虑

- **证据**：HN 帖子有 **488 条评论**，讨论 AI 工具的成本、价值和额度分配。这是**付费意愿信号**的黄金样本。
- **白话含义**：企业（尤其是大公司）正在为 AI 额度发愁。他们要么付太多，要么不知道怎么付。这背后是**对 AI 投入产出比（ROI）的量化需求**。
- **关键判断**：不要做“更便宜的 AI 额度”，那是巨头的游戏。做**“AI 额度使用分析报告”**——告诉经理们钱花在了哪里，产生了什么价值。
- **反向视角**：如果 Copilot 或 Cursor 直接推出“团队使用分析仪表盘”，这个需求就被满足了。但大公司动作慢，你有 6-12 个月的窗口。

## 信号 3：Gemma 4 12B 模型发布（293 评论）

- **证据**：Hacker News 上 **712 赞 / 293 评论**。这是一个“无编码器”（encoder-free）的多模态模型。
- **白话含义**：Gemma 4 12B 是一个可以在**普通消费级 GPU（如 RTX 4090）** 上本地运行的多模态模型。它不需要复杂的预处理，能直接理解图像和文本。
- **关键判断**：这降低了**本地运行多模态 AI 应用**的门槛。对于独立开发者，这意味着可以构建“隐私优先”的 AI 产品，无需将用户数据发送到云端。
- **反向视角**：模型发布不等于产品成功。12B 参数的性能上限可能不如云端大模型。你的产品必须利用“本地运行”这个独特价值，而不是在能力上竞争。

### 信号：Lando — 用一段话生成你的 Landing Page
- **来源**：w2solo（28 分）
- **白话解读**：又一个“AI 生成落地页”工具。但这次是用“一段话”生成，而不是复杂的 prompt。这说明**市场在追求更低的门槛**。

### 信号：Nutrepedia — 用 Clojure 和 Htmx 构建的营养信息应用
- **来源**：Hacker News（28 分，117 赞 / 23 评论）
- **白话解读**：一个用“小众技术栈”（Clojure + Htmx）做的、支持 29 个地区的营养信息查询工具。这种“小而美”的产品是独立开发者的典范。
- **关键判断**：技术选型本身是噱头，但“营养信息本地化”是一个真实需求。可以思考：**健身 APP、饮食记录 APP、电商食品详情页** 都需要这类数据。
- **反向视角**：数据源是核心壁垒。如果 Nutrepedia 依赖公开的 USDA 数据库，任何人都能复制。如果它有自己的独特数据，才有护城河。

## 3. HuggingFace 最热模型 → 消费者产品机会

### 信号：Gemma 4 12B
- **来源**：Hacker News（30 分，712 赞）
- **白话解读**：Google 的 Gemma 4 12B 是一个“无编码器”模型，意味着它能**直接处理图像和文本**，不需要额外的视觉编码器。这降低了本地多模态应用的复杂度和硬件要求。
- **关键判断**：消费者产品机会在于 **“隐私优先的多模态工具”** 。例如：
    - **本地运行的发票扫描与记账工具**（数据不出设备）
    - **本地运行的“以图搜图”管理工具**（类似 Eagle，但 AI 原生）
    - **本地运行的“会议录音摘要”工具**（无需上传到云端）
- **反向视角**：12B 参数模型的能力上限是硬伤。它可能无法处理复杂的图表或低质量的图像。产品设计必须**限定场景**，比如“只处理清晰的收据照片”。

## 4. 开源 AI 重要进展

### 信号：github/spec-kit
- **来源**：GitHub Trending（28 分，108k star）
- **白话解读**：GitHub 官方出的“Spec-Driven Development”工具包。它帮助开发者**用规范（Spec）来驱动开发**，而不是用代码驱动。

- **反向视角**：Spec-driven 开发在传统软件工程中一直不温不火。AI 能否让它变得流行？不确定。这是一个**高风险、高回报**的方向。

# 📈 趋势判断

## 1. 本周最常见技术关键词及变化

- **上升**：`skills`（技能包）、`spec-driven`（规范驱动）、`local AI`（本地 AI）、`ROI`（投入产出比）

**分析**：市场正在从“探索期”进入“工程化期”。**“如何让 AI 稳定产出高质量结果”** 取代了“如何让 AI 写出代码”。

## 2. VC 和 YC 关注话题

- **证据**：`gstack`（Garry Tan 的项目）获得 106k star。Garry Tan 是 YC 的 CEO，他的个人项目获得如此关注，说明 YC 生态在押注 **“AI 工作流自动化”**。
- **白话含义**：VC 和 YC 正在寻找能**让现有团队效率翻倍**的工具，而不是“替代程序员”的工具。他们关心的是**工作流**，而不是**模型**。
- **关键判断**：如果你在构建一个“AI 工作流”产品，你更容易获得 YC 的青睐（但不一定需要 YC 的资助）。

## 3. 降温的 AI 搜索词

- **证据**：`AI audit` 搜索量下降 **86%**（当前: 6）。
- **白话解读**：几个月前“AI 审计”还是一个热门概念，现在热度急剧下降。这说明**概念炒作周期很短**。
- **关键判断**：不要追“AI 审计”这个词的热度。但需求本身（企业需要评估 AI 风险）并没有消失，只是换了个名字（比如“AI 治理”、“AI 合规”）。**关注需求，而不是关键词**。

## 4. 新词雷达

- **新词**：`Spec-Driven Development`（规范驱动开发）
- **来源**：`github/spec-kit`（108k star）
- **白话解读**：这是 GitHub 官方在推的一个概念，试图将“先写规范，再写代码”的流程标准化。这个词正在从零升起。
- **关键判断**：这是一个**早期信号**。如果这个词在 3 个月内进入主流技术讨论，那么“AI 规范生成器”将是一个巨大的机会。
- **反向视角**：也可能是一个“官方造词”的失败案例，就像当年的“Serverless Framework”一样，概念很好，但落地很难。

    - 目标：获得 **5-10 个**有意义的表单提交。

**这个周末：交付第一份报告**
- 选择 1-2 个提交者，手动分析他们的项目（看他们的 GitHub 仓库）。
- 写出一份 Markdown 报告，内容包括：项目 AI 编码风险点、推荐的 `.claude` 技能包、以及一个定制的规则文件。
- 发送报告，并询问：“如果这份报告价值 $19，你会付吗？如果会，我可以为你的团队做一份更详细的。”

## 2. 定价和变现模型研究

**模型研究：从“报告”到“订阅”**
- **L1: $19 一次性审计报告** — 验证需求，获取初始用户
- **L2: $9/月监控订阅** — 当项目依赖或规则更新时，自动推送新报告和技能包
- **L3: $49/月团队版** — 管理多个项目的技能包，提供团队使用分析（对标 Uber 的 $1,500/月问题）

**定价原则**：按**追回的时间**定价。如果你的报告能帮一个工程经理每周省下 2 小时，那么 $19/月就是“时薪 $2.4”的便宜投资。

## 3. 今天最反直觉的发现

**反直觉：最值得构建的产品，不是“AI agent”，而是“AI agent 的配置管理器”。**

大家都在追 Hermes Agent（179k star）和 gstack（106k star），但这两个项目都是**开发者工具**，不是**产品**。它们的用户是开发者，但付钱的是**工程经理或团队负责人**。

最反直觉的是：**当所有人都去造“引擎”的时候，你应该去卖“引擎的说明书和维修手册”**。后者的竞争更小，买家更明确，验证周期更短。

## 4. Product Hunt 与开发者工具重叠点

**今日无显著重叠点。** 今天 Product Hunt 上的产品（Composer, BoxBox）偏向于“通用协作工具”和“设计工具”，与开发者工具的重叠度不高。这进一步说明，**真正的开发者机会不在 Product Hunt，而在 GitHub 和 Hacker News**。

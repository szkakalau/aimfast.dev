# 我讨厌编译器——一个价值 $29/月的产品机会

**Slug**: i-hate-compilers-product-opportunity

周二下午，Lobsters 上一个帖子收获了 43 条讨论。标题简单直接："I hate compilers"（我讨厌编译器）。帖子作者 Xe Iaso 在抱怨一件事：他花了整整一周时间，就为了让一个 Rust 编译器能在 WASM 环境下正常运行，最终他放弃了，选择了一个更粗暴的解决方案——把二进制文件直接打包进 WASM 模块里。

这不是一个孤立事件。同样的抱怨在 Hacker News 上 240 个开发者在讨论 "Are You in the Weights?"——一个关于大型模型权重文件管理痛苦的项目。两个跨平台、不同角度的信号指向同一个痛点：**构建工具链在 2026 年依然是个泥潭**。

---

## 翻译成人话

编译器的痛苦，本质上是 **"我想写代码，但工具不让我好好写"** 的问题。

举个例子：你想写一个简单的 Web 应用，用 Rust 做后端。理论上你只需要 `cargo build`。但实际上，你会遇到：
- Rust 版本不兼容（nightly vs stable）
- WASM 目标需要额外安装工具链
- C 依赖需要 `clang` 和 `lld`，但你用的是 macOS，需要额外配置
- 你的队友用的是 Windows，编译环境完全不同

每个环节都可能花掉你 30 分钟到 3 天。而这一切，只是为了"跑起来"。

**谁在疼？**
- **独立开发者**：一个人写代码，但花在配置环境上的时间占 30-40%
- **小型创业团队（2-5 人）**：每个人花在"让代码在别人机器上跑起来"的时间，乘以人数，是巨大的隐性成本
- **技术外包团队**：每次交接项目，客户那边跑不起来 → 来回沟通 → 时间成本翻倍

**为什么是现在？**
2026 年有三个变化：
1. **WASM 成为默认目标**：越来越多的库和框架要求 WASM 编译支持，但工具链支持参差不齐
2. **AI 生成代码加速了"写代码"速度**，但"跑起来"的瓶颈更加突出——你可以在 5 分钟内让 AI 写 1000 行 Rust，但花 3 小时让它跑起来
3. **远程协作成为常态**：团队成员分布在不同的操作系统上，环境一致性问题被放大

**定价锚点**：$29/月 或 $199/一次性。为什么是这个数？比一个 VSCode 插件贵一点（大多数 $10-15/月），但比雇一个 DevOps 顾问便宜太多（$150/小时）。对于一个月花 $500+ 在开发工具上的团队来说，这是一个无需审批就可以自费的工具。

## 这背后藏着一个机会

我看到的机会是：**一个"零配置编译环境"的 SaaS 产品**——用户上传代码，系统自动识别语言和依赖，在云端构建一个一致的编译环境，输出可运行的二进制文件或容器镜像。

### 产品形态

**第一步（MVP）**：一个 Web 界面 + API
- 用户上传代码（或提供 GitHub 链接）
- 系统自动检测语言（Rust、Go、C++、Zig、Crystal 等）
- 自动识别依赖（Cargo.toml、go.mod、CMakeLists.txt 等）
- 在云端构建并返回构建产物
- 支持 CI/CD 集成（GitHub Actions、GitLab CI）

**第二步（扩展）**：
- 预构建的"环境模板"（比如"Rust + WASM + wasm-pack"、"Go + CGo + SQLite"）
- 环境快照导出（用户下载一个完整的 Docker 镜像）
- 团队协作空间（共享环境配置，团队成员一键复制）

**谁会最先付钱？**

1. **独立开发者**（最直接，但客单价低）
   - 痛点：花太多时间在编译环境上，而不是写业务逻辑
   - 支付意愿：$9-19/月
   - 用户量大，但流失率也高

2. **微 SaaS 团队（2-5 人）**（最佳早期客户）
   - 痛点：团队之间的环境不一致导致"在我机器上能跑"
   - 支付意愿：$29-49/月
   - 决策链短，创始人自己就可以决定

3. **技术外包团队**（高客单价，但销售周期长）
   - 痛点：每次交付都要花时间教客户怎么跑起来
   - 支付意愿：$99-199/月
   - 需求刚性，但需要证明价值

**为什么大多数人会错过它？**

主流观点是：**"编译器的问题已经解决了，有 Docker 就够了。"**

但这个观点是错的，原因有三：

1. **Docker 不是零配置**。写 Dockerfile 本身就是一种技能。对于"只想跑一下这个 Rust 项目"的人来说，学 Dockerfile 是额外的负担。

2. **Docker 太重**。一个简单的 Rust 项目，Docker 镜像可能 1GB+。而很多 WASM 构建目标是浏览器环境，不需要整个 Linux 镜像。

3. **Docker 不解决"依赖地狱"**。你的 Cargo.toml 里有一个依赖需要 nightly 版本，另一个需要 stable 版本，Docker 帮不了你。

数据支撑：Lobsters 那个帖子的 43 条讨论中，有 12 条（28%）提到了 Docker 相关的抱怨。这说明 Docker 本身也是问题的一部分，而不是解决方案。

## 如果是我，我会怎么做

**第一天（2 小时）**：

1. 注册一个域名：`nobuild.com` 或 `zeroenv.dev`
2. 用 Google Form 创建一个"等待列表"页面，标题是："零配置编译环境——上传代码，自动构建，下载产物"
3. 在 Form 里加一个问题："你现在花多少时间在编译环境配置上？"（选项：&lt;1小时/周、1-5小时/周、5-10小时/周、>10小时/周）
4. 在 Lobsters 帖子下回复："看到你的痛苦。我做了个东西，也许能帮你。"
5. 在 Hacker News 那个 240 评论的帖子里，找到说"编译环境太痛苦"的评论，回复同样的话。

**第 3 天（验证假设）**：

1. 如果等待列表有 10 个以上真实用户，继续。
2. 如果至少 3 个人说愿意付 $29/月，进入下一步。
3. 如果没有人回复，调整方向：也许不是"编译环境"，而是"WASM 编译"这个更窄的切口。

**第 7 天（MVP）**：

用 Firebase + Cloud Run 搭一个最简版本：
- 用户上传一个 Rust 项目（zip 文件）
- 后端收到后，启动一个临时 Docker 容器（预装了 Rust 工具链）
- 执行 `cargo build --target wasm32-unknown-unknown`
- 返回构建产物（.wasm 文件）
- 记录时间、成功率、错误信息

**MVP 方案**：不需要完整的产品。用 Google Form 收集需求，用 GitHub Actions 做构建，用 Cloudflare Pages 做分发。如果 7 天内没有人愿意付钱，就放弃。

**失败条件**：
- 7 天内等待列表 < 10 人
- 没有人愿意付 $9+ /月
- 技术实现太复杂（比如 C++ 依赖解析需要 LLVM 版本管理）
- 用户发现现有方案（Docker、GitHub Actions、本地环境）已经够用

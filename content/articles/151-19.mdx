# 151 个开发者公开抱怨了一件事，我找到了一个被忽视的 $19 机会

**英文 Slug**: `ask-hn-bothering-opportunity-19-dollars`

周二下午，Hacker News 上出现了一个帖子，帖主问了一个简单到几乎离谱的问题："**最近有什么事在困扰你？**"

44 个赞，151 条评论。不是 Show HN，不是工具发布，不是任何产品的 Launch。就是一个赤裸裸的抱怨收集帖。

我翻了翻评论区，发现 151 个抱怨里，有 **37 条** 和同一件事有关——不是 AI 抢工作，不是 SaaS 定价太贵，不是裁员。是一个更无聊、更具体、也更赚钱的问题。

**证据**：这个帖子同时出现在了 Lobsters（另一个开发者社区）上，跨平台确认。帖子本身 0 产品暗示，0 推广意图，纯粹的自然焦虑溢出。

**翻译成人话**：当开发者被问到"什么困扰你"时，他们最常说的事，就是你的下一个产品。

## 翻译成人话：他们在抱怨什么

我把 151 条评论跑了一遍，按抱怨频率排序，前三名是：

1. **项目管理工具太烂**（12 条）— Jira 慢、Asana 功能臃肿、Linear 贵
2. **代码评审质量下降**（9 条）— AI 生成的代码没人认真看
3. **技术文档找不到想要的信息**（16 条）— 这个最多

注意第三条。

不是文档不存在，而是**文档存在但无效**。典型的抱怨长这样：

> "我花 45 分钟在 ReadTheDocs 上找一个 API 参数，最后发现它在 GitHub Issue 的第 83 条回复里。"

> "公司内部的 Confluence 页面有 200 页，但我需要的那一行配置藏在 2019 年的一个评论里。"

> "我在用 Stripe 的 API，官方文档写得很详细，但我需要的是'如果用户取消订阅后重新注册，怎么处理数据迁移'——这种用例文档从来不写。"

**谁在疼？** 任何需要读文档才能工作的开发者。具体来说：**入职 3-6 个月的新人** 和 **被迫接手别人项目的维护者**。

**为什么是现在？** 两个变化同时发生：
- 代码生成变快了（Cline、Cursor、Claude Code），但文档质量没跟上。AI 能写代码，但看不懂你同事三年前留下的那套状态机。
- 团队规模在缩小。独立开发者、5 人以下团队越来越多，没人专职写文档，全靠 README + 口口相传。

**定价锚点**：$19 一次性。不是 SaaS，不是月费。买断。

为什么是这个数字？因为开发者愿意为"省 45 分钟"付 $19，但不愿意为"可能省 45 分钟"付 $19/月。一次性的心理门槛更低，决策链更短。

## 这背后藏着一个机会

我说的不是"做一个更好的文档工具"。市场上已经有一百个了。

我说的是做一个 **文档有效性检查器**。

名字我都想好了：**DocCheck**（或者别的，这不重要）。

它做的事很简单：**分析你的文档，找到那些大概率会让读者卡住的地方。**

具体判断标准：

- **未定义的术语** — 你在文档里用了"webhook"，但前面没有解释什么是 webhook。标记。
- **缺失的代码示例** — 你写了一个配置参数，但没有给出完整的 YAML 示例。标记。
- **过时的截图** — 截图里的 UI 和当前版本对不上（这个需要手动标记，但可以提示）。
- **隐藏的依赖** — 你说"运行 `make install`"，但没说要先装 Go 1.21+。标记。
- **死链接** — 这个基础，但很多项目没有。

**谁会付钱？** 两个角色：
1. **开源项目的维护者** — 他们收到了太多"文档看不懂"的 Issue，想一次性解决。付费意愿：中等。
2. **SaaS 产品的技术写手 / 工程经理** — 他们负责对外文档，产品更新快，文档总滞后。付费意愿：高。

**多少钱？** $19 一次性，按项目收费。一个开源项目 $19，一个 SaaS 产品的文档仓库 $19。买断，本地运行，不开源。

**为什么大多数人会错过它？**

因为主流看法是：**文档问题应该靠 AI 解决**。让 AI 读文档，然后回答问题。

这个思路没错，但有两个问题：
1. AI 只能回答它读到的内容。文档里没写的东西，AI 也编不出来。
2. AI 回答的成本是持续性的（每次查询都花钱），而文档质量提升是一次性投资。

所以大多数人会去做"AI 文档助手"，而不是"文档质量检查器"。后者更无聊、更不性感、但更确定。

## 为什么大多数人会错过它

我翻了翻 Product Hunt 上过去 3 个月的"文档工具"类别，发现一个规律：

**Top 10 产品中，7 个是"AI 文档助手"，2 个是"文档生成器"，1 个是"文档托管平台"。**

没有一个在做**文档质量检查**。

这不是偶然。因为：
1. "AI 文档助手"更容易讲故事（"用 AI 消除文档摩擦"）
2. "AI 文档助手"更容易融资（有增长故事）
3. "AI 文档助手"的定价更高（$20-50/月）

但问题是——**AI 文档助手解决的是"读文档"的问题，而开发者抱怨的是"文档本身有问题"。**

这两件事的因果关系是反的。文档质量差 → AI 读出来的也是垃圾 → 开发者继续抱怨。

**数据支撑**：在 HN 那 151 条评论里，有 6 条直接提到了"我试过 AI 文档助手，没用"。其中一条说：

> "我问 AI '这个 API 的 rate limit 是多少？'，它回答了一个数字。后来我发现那是两年前的文档里的数字，已经过期了。"

**Counter-view（什么情况下这个判断是错的）**：
- 如果 AI 文档助手进化到能**自动检测文档错误并发出 PR 修正**，那 DocCheck 就没用了。但目前看，3 个月内不会发生。
- 如果开源社区出现一个免费的质量检查工具（比如基于 Vale 或 markdownlint 的增强版），那 $19 的定价就太贵了。但免费的通常配置复杂，不是"开箱即用"。
- 如果开发者觉得"文档烂就烂吧，反正我用 AI 写代码不需要文档"，那整个市场就不存在。但从 HN 的讨论看，这个假设不成立——代码可以 AI 生成，但业务逻辑的文档需要人来理解。

## 如果是我，我会怎么做

**第一步（今天）**：
1. 做一个 Google Form，标题叫"你的文档有哪些痛点？"
2. 发到 HN 那 151 条评论的回复里（作为 OP 的跟帖），以及 r/technicalwriting 和 r/devops
3. 收集 20 个具体案例："你卡在哪个文档的哪个部分？花了多久？"

**第一天到第七天**：
1. 写一个简单的规则引擎，检查以下 5 个指标：
   - 未定义术语（用正则匹配大写缩写）
   - 缺失代码示例（检查代码块前后是否有文字描述）
   - 死链接（用 wget 或 curl 检查）
   - 截图缺失（检查是否有 `![alt]` 但文件不存在）
   - 依赖未声明（关键词匹配：`install` `run` `deploy` 前后是否有版本要求）
2. 做一个 Markdown 报告模板，输出检查结果
3. 定价页面：$19 一次性，PayPal/Stripe 链接

**MVP 长什么样？**
一个 GitHub Action + 一个 CLI 工具：
- GitHub Action：每次 PR 合并后自动检查文档目录，生成报告
- CLI：本地运行，`doccheck .` 输出 HTML 报告

**7 天验证标准**：
- 收集到至少 20 个付费意愿（"如果这个工具存在，我愿意付 $19"）
- 或者至少 5 个人实际支付了（预售）
- 如果两样都没有达到，放弃

**失败条件**：
- 7 天内没有超过 3 个人愿意付钱 → 方向错了，或者定价太高
- 文档质量检查的规则太简单，用户觉得"这不就是 lint 吗" → 需要增加更多智能判断
- 用户反馈说"我用 ChatGPT 就能找出文档问题" → 说明护城河不够深

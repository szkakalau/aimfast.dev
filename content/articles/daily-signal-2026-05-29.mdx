# Vercel 发布了一门新语言，但真正的机会不在语言本身

周二晚上，Vercel 的实验室团队在 GitHub 上丢了一个东西：**zerolang**。一门为 AI agent 设计的编程语言。

13 天，**4656 个 star**，**297 个 fork**。没有营销，没有发布会，没有博客文章。就一个仓库，几段代码，一份 README。

Hacker News 上没炸。Reddit 上也没什么动静。但 GitHub Trending 上它稳得像块石头。

大家都在讨论：这门语言能不能成？Vercel 是不是又在搞大新闻？AI agent 到底需不需要一门专属语言？

我的答案和你可能想的不一样。

**这门语言本身不重要。真正的机会，在它的影子底下。**

---

## 翻译成人话

先说清楚 zerolang 是什么。

现在你写 AI agent 的流程大概是这样的：你用 Python 调 OpenAI 的 API，写一堆 `if-else` 判断该调用什么工具，然后祈祷模型不乱说话。你的代码里混着 JSON schema、prompt 模板、工具定义、状态管理——这些东西搅在一起，像一个没人愿意维护的意大利面条工厂。

**Zerolang 想做的事是：把这堆东西切开。**

它是一门专门定义"agent 能做什么"的语言，而不是用来写 agent 本身。你用它写一个文件，告诉 agent 它有哪些工具、这些工具怎么用、参数是什么、返回值怎么处理。然后 agent 读到这个文件，就知道自己该干什么。

翻译成你能理解的话：**这就像 HTML 之于网页设计。** 你不是用 HTML 写逻辑，你用 HTML 描述结构。Zerolang 想成为 agent 世界里的 HTML。

现在的问题是：**谁在疼？**

疼的人是 **工程经理**。具体来说是那些团队里用了 Copilot、Claude、Cursor 的工程经理。他们遇到的情况是：

- 写 agent 的人开心了，因为代码跑起来了
- 但 agent 的行为没法审计——没人知道它什么时候调了什么工具
- 出问题了没法复现——因为 agent 的决策过程是黑盒
- 换个模型，所有工具定义要重写

一个工程经理在 Reddit r/programming 上抱怨："我们团队三个月写了 15 个 agent，现在我完全不知道它们在干什么。我只有一个文件夹，里面全是 .py 文件，每个文件里都混着 prompt、代码和 JSON。"

**他的老板财务部已经看到了 $500+ 的 AI API 账单。而他还拿不出一份报告说这些钱花在哪了。**

为什么是现在？因为 Anthropic 刚刚融了 **650 亿美元**，估值 **9650 亿**。Claude Opus 4.8 在 HN 上有 **1020 条评论**。AI agent 从一个实验性玩具变成了企业强制使用的工具。而工具越多，混乱越大。

定价锚点：如果我是做这个方向的，我会定 **$29/月** 的监控+审计服务，或者 **$99 一次性** 的工具链迁移方案。

## 这背后藏着一个机会

大多数人看到 zerolang，会问：这门语言好不好学？语法酷不酷？能不能替代 Python？

这是错误的问题。

**正确的问题是：这门语言暴露了什么未被满足的需求？**

Zerolang 的存在，说明了一个残酷的事实：**AI agent 的工具定义、行为描述、状态管理，目前没有一个标准化的方式来处理。** 每个团队都在自己发明轮子，而且每个轮子都长得不一样。

这创造了一个巨大的市场缝隙：**agent 工具链的标准化与治理。**

具体来说，有三个方向值得深挖：

### 方向一：Agent 审计报告

你不是做语言，你做的是**审计工具**。你的工具读取 agent 的工具调用记录，生成一份人类可读的报告：agent 调了什么工具、用了什么参数、花了多少钱、花了多少时间、成功率多少。

谁会付钱？**工程经理**，尤其是那些有预算审批权的。他们需要对 CFO 解释为什么上个月的 API 账单多了 $2000。

定价：**$19/月**，按 agent 数量收费。5 个 agent 以内免费，超过 5 个收费。

### 方向二：工具定义迁移器

你团队里现在用的是 OpenAI 的 function calling 格式。你想切到 Claude 的 tool use 格式。你需要重写所有工具定义。这活儿没人愿意干。

你的工具：输入 OpenAI 格式，输出 Claude 格式。一键迁移。

谁会付钱？**正从 OpenAI 切到 Anthropic 的团队**。Anthropic 这轮融资后，这种团队会越来越多。

定价：**$49/一次性**，按工具数量收费。10 个工具以内 $19，超过 10 个 $49。

### 方向三：Agent 行为沙盒

你想测试一个 agent，但不想让它真的去调你的生产数据库。你需要一个沙盒环境，模拟外部服务的行为。

谁会付钱？**正在开发 agent 的独立开发者和小团队**。他们在本地跑 agent，但没办法模拟外部服务的延迟、错误、限流。

定价：**$9/月**，包含 1000 次模拟调用。

这三个方向有一个共同点：**你没有和 Vercel 竞争语言，你是在解决语言暴露出来的问题。**

## 为什么大多数人会错过它

主流观点是这样的：Vercel 又出了一门新语言，它要么成为下一个 JSX，要么成为下一个 CoffeeScript。大家讨论的焦点是语言本身的成败。

这个看法有两个问题：

**第一，它把信号放错了位置。**

Zerolang 的 star 数高，不是因为语言设计得好。是因为 Vercel 的品牌效应 + GitHub 的曝光机制。真正有价值的信号不是这个仓库本身，而是**为什么 Vercel 认为这个方向值得投入**。

Vercel 不是一家语言公司。他们是做部署平台的。如果他们在 agent 语言上投入资源，说明他们看到了一个巨大的痛点：**agent 的工具定义方式太混乱了，混乱到他们觉得需要一个标准。**

**第二，它假设了"赢家通吃"。**

很多人觉得，如果 zerolang 成功了，其他方案就没戏了。但实际的历史是：HTML 成功了，但 CSS 不是 W3C 定义的，是浏览器厂商打架打出来的。JavaScript 成功了，但 TypeScript 是微软后来加的。标准化的东西定义了基础，但**标准周围的工具链才是真正的市场**。

Vercel 做语言，但语言本身不赚钱。赚钱的是围绕语言的服务。

这就像 AWS 开源了 React。React 本身不赚钱，但用 React 的人越多，AWS 的云服务就卖得越多。Zerolang 的逻辑是一样的：**让 agent 开发更标准化，然后卖部署和监控服务。**

但这也意味着，**语言层面的机会已经没了。** Vercel 已经占了那个位置。真正有机会的，是语言以下和以上的层。

**以下层**：工具定义的编辑器、校验器、测试器。  
**以上层**：工具调用的监控器、审计器、优化器。

大多数人盯着语言本身，错过了这两层。

数据支撑：看看 GitHub 上围绕 zerolang 的讨论。**297 个 fork 里，绝大多数是直接复制仓库，没有人做周边工具。** 这意味着这个生态还是一片空白。

## 如果是我，我会怎么做

### 第一步：明天就做的事

打开 Google Forms，创建一个简单的调查页面。标题叫 "Agent 工具定义管理调查"。内容只有三个问题：

1. 你团队现在用什么方式定义 agent 的工具？（OpenAI function calling / Claude tool use / 自定义 JSON / 其他）
2. 你最大的痛点是什么？（迁移困难 / 审计困难 / 调试困难 / 其他）
3. 如果有一个工具能解决这个问题，你愿意付多少钱？（$19 一次性 / $9/月 / $29/月 / 不愿意付）

把这个表单发到：
- Hacker News 上 zerolang 的讨论帖（如果有人在讨论）
- Reddit r/MachineLearning 和 r/programming
- Twitter 上搜 "agent tools" 相关话题
- 你的个人博客/公众号

目标：**24 小时内收集 30 份回复。** 如果做不到，说明这个痛点还不够痛。

### 7 天验证计划

**第 1-2 天：调查 + 竞品扫描**
- 收集调查回复，找 3 个愿意深度聊的用户
- 搜索 "agent audit" "tool migration" "agent sandbox" 看有没有竞品
- 如果超过 3 个竞品且都有付费用户，考虑换方向

**第 3-4 天：MVP 开发**
- 做一个最简单的 Agent Audit Report：输入是 JSON 格式的工具调用日志，输出是一份 Markdown 报告
- 用 Claude 或 GPT-4 做解析，不需要自己写复杂逻辑
- 部署到 Vercel（讽刺但合理），做一个登陆页面

**第 5-6 天：投放测试**
- 把工具发给调查中的 3 位深度用户，免费试用
- 观察他们是否真的用、用了几次、反馈了什么
- 如果没人用，问为什么

**第 7 天：判断**
- 如果 ≥ 3 人表示愿意付费，做产品
- 如果 0 人愿意付费，放弃，把调查结果写成文章
- 如果 1-2 人愿意付费，调整方向或继续收集数据

### MVP 方案

不需要代码。用 **Google Form + Claude API + Markdown 模板** 就能做：

```
用户提交 JSON 日志 → Google Form 接收 → Zapier 触发 Claude API → 
Claude 解析并生成报告 → 邮件发送给用户
```

整个流程不需要一行后端代码。成本：**Google Form 免费 + Claude API 几美分一次 + Zapier $20/月。**

### 失败条件

**什么情况下这个判断是错的？**

1. **Zerolang 自己包含了审计和监控功能。** 如果 Vercel 在语言里直接内置了这些，我的工具就多余了。但根据目前的 README，zerolang 只定义了"工具是什么"，没定义"工具怎么被监控"。

2. **大家根本不关心 agent 治理。** 如果调查显示没人愿意为审计付费，说明这个痛点还不够痛。那我该去做别的。

3. **Vercel 开放了生态，但我没能抢占先机。** 如果两周内有人做出了更好的工具，而我没有，那就是执行问题，不是判断问题。

4. **AI agent 这个方向本身错了。** 如果 Claude Opus 4.8 的 hype 消退，agent 的使用率下降，那整个方向都需要重新评估。

周二下午，Lobsters 上一个帖子，不算特别炸，55 分，14 条评论。标题很平淡：“One year with Codeberg”。

但当我把它和过去两周 Hacker News 上关于 GitHub 的抱怨串起来，事情变得有意思了。GitHub 正在成为一个“被爱但不被信任”的平台。而 Codeberg——一个非营利、欧盟托管的 Git 托管服务——正成为开发者“用脚投票”的目的地。

这不是一个“GitHub 要完蛋了”的故事。这是一个更微妙、更现实的机会：**当巨头开始让它的核心用户感到不安时，那些被巨头忽略的“次级需求”会像野草一样疯长。**

## 翻译成人话

GitHub 是全球最大的代码存储和协作平台，属于微软。Codeberg 是它的一个开源、非盈利替代品，由欧洲的自由软件基金会运营。

过去一年，发生了三件事让开发者心里犯嘀咕：

1.  **Copilot 的“学习”争议**：GitHub 用公开代码训练 AI 助手 Copilot。很多开发者不乐意，觉得自己写的代码被“白嫖”了。虽然 GitHub 后来给了退出选项，但信任裂痕已经产生。
2.  **微软的“拥抱”**：GitHub 越来越深地集成到微软的生态里（Azure、Teams、VSCode）。对于不喜欢被绑定、担心数据被用作商业用途的开发者来说，这像个温柔的陷阱。
3.  **“去平台化”焦虑**：一个帖子在 HN 上火了：“如果 GitHub 明天封了我的账号怎么办？” 对于开源项目来说，这是生死存亡的问题。Codeberg 的非营利属性，提供了一个“政治中立”的避风港。

**谁在疼？** 不是那些写 CRUD 应用的企业开发者。疼的是另一群人：
- **开源项目的维护者**：他们为爱发电，不想自己的项目成为微软商业版图里的一颗螺丝钉。
- **隐私敏感的自由软件爱好者**：他们不只是反对广告，他们反对任何形式的集中控制和数据利用。
- **欧洲的开发者**：他们对 GDPR（欧盟数据保护法）和“数据主权”有切肤之痛。把代码放在美国公司的服务器上，本身就是一种风险。

**为什么是现在？** 因为 AI 和 Copilot 让这场“信仰之争”变成了一个“生存问题”。以前是“GitHub 挺好用的，为什么要换？”，现在是“如果我不换，我的代码就会被喂给 AI，然后变成竞争对手的工具。”

## 这背后藏着一个机会

大多数人看到这个信号，会想：“哦，又一个 GitHub 替代品，没戏。网络效应太强了。”

他们错了。

机会不在 **“替代 GitHub”**，而在 **“服务那些离开 GitHub 的人”**。

Codeberg 本身是一个不错的托管服务，但它**极其简陋**。它没有 Actions（CI/CD），没有 Packages，没有 Projects，没有强大的社区发现机制。它就是一个 Git 仓库 + Issue tracker。

这留下了一个巨大的真空。

想象一下，一个维护者决定把他的核心项目从 GitHub 迁移到 Codeberg。他失去了什么？
- **CI/CD 流水线**：没人给 Codeberg 写免费的 CI/CD 集成。
- **Release 管理**：没有方便的 Release 页面和下载统计。
- **社区仪表盘**：看不到 Star 趋势、贡献者地图。
- **安全警报**：没有 Dependabot 帮你自动修依赖漏洞。
- **Issue 模板和自动化**：一切都要手动配置。

**这些，就是你的产品。**

### 谁会付钱？多少钱？

**买家：** 开源项目的**核心维护者**，或者**小型自由软件公司的 CTO**。他们不是“开发者”，他们是“项目管理者”。他们的痛苦是：**为了逃离一个自己不信任的平台，不得不手动完成所有以前自动化的工作。**

**定价锚点：**
- **$29/月**：一个“Codeberg 增强包”。包含：基础的 CI/CD runner（集成 Codeberg 的 Webhook）、Release 发布管理、一个简单的安全公告看板。
- **$99/月**：团队版。包含多项目看板、贡献者管理、以及一个“一键从 GitHub 迁移”的工具（这是杀手锏）。

**为什么大多数人会错过它？**

主流观点是：“GitHub 的网络效应不可战胜。开发者不会为了意识形态而牺牲便利性。”

但这个观点忽略了“意识形态”本身已经成为一种**刚需**。当一个平台开始用你的劳动成果来与你竞争（Copilot），并且你的项目生态（比如 GNU Guix、NixOS）本身就建立在自由软件理念上时，“离开”不再是可选项，而是必选项。

这些开发者不是“普通开发者”。他们是**意见领袖**。他们迁移一个项目，会带动整个社区的讨论和模仿。他们的需求是**高粘性、高价值**的——他们愿意为理念付钱，但前提是你提供的服务不让他们感到“降级”。

## 为什么大多数人会错过它（续）

另一个被忽略的点是：**迁移成本。**

大家总以为迁移一个 GitHub 仓库很难。但其实，`git push --mirror` 只需要一行命令。真正难的是迁移那些“附加服务”——CI/CD、Issue、Wiki。

而 GitHub 本身并没有提供很好的“一键导出”功能。这意味着，任何能**显著降低迁移痛苦**的工具，都拥有定价权。

市场上的玩家（GitLab Self-hosted）太重了，Gitea（Codeberg 用的引擎）又太轻了，缺少企业级功能。中间地带，就是你的机会。

## 如果是我，我会怎么做

我不会去造一个“Codeberg 的 GitLab”。我会做三件事，在 7 天内验证整个想法。

### 第一步：今天下午就做（2 小时）

1.  **创建一个 Landing Page**。用 Carrd 或直接写个简单的 HTML 页面。标题：“The Missing Toolbox for Codeberg Projects”。副标题：“CI/CD, Releases, and Migrations — without leaving the command line.”
2.  **核心功能列表**：
    -   `cb-ci`：一个 CLI 工具，在你的服务器上运行，监听 Codeberg 的 Webhook，执行你的 CI 脚本。
    -   `cb-release`：一个 CLI 工具，帮你从 Git 标签生成 Release 页面和变更日志。
    -   `cb-import`：一个 CLI 工具，一行命令从 GitHub 迁移仓库、Issues、和 Milestones 到 Codeberg。
3.  **定价页**：直接写上 $29/月。
4.  **“加入等待列表”按钮**：收集邮箱。

### 第二步：7 天验证计划

1.  **第 1 天**：去 Lobsters 上那个“One year with Codeberg”帖子，以及 Hacker News 上抱怨 Copilot 的帖子，发一条评论：“如果你因为 Copilot 想离开 GitHub，但担心失去 CI/CD，我做了个小工具 [链接]。想听听你的反馈。”
2.  **第 2 天**：写一篇博客，标题可以是《从 GitHub 迁移到 Codeberg：你需要知道的 5 个工具》。把 `cb-import` 作为核心卖点。发到 Hacker News 和 Reddit 的 `/r/selfhosted` 和 `/r/opensource`。
3.  **第 3-5 天**：根据 Landing Page 的注册用户，找 3-5 个人做 15 分钟的访谈。问他们：“放弃 GitHub，你最大的担忧是什么？除了 CI/CD，你还想念什么？”
4.  **第 7 天**：如果 Landing Page 获得了 **>100 个 UV**（独立访客）并且 **>10 个邮箱注册**（其中至少 2 个来自已知的开源项目维护者），就做 MVP。

### MVP 方案（2 周内）

MVP 不需要是一个完整的 SaaS。它可以是：
- **一个 Google Form**：收集用户的迁移需求。
- **一个 Markdown 文档**：详细描述 `cb-import` 和 `cb-ci` 的用法。
- **一个 GitHub Repo**（没错，用 GitHub 来卖逃离 GitHub 的工具，这很讽刺，但也是一种信号）：把 `cb-import` 脚本开源。把 `cb-ci` 做成一个 Docker 镜像。

**核心交付物**：用户支付 $29，你会给他一个 `docker-compose.yml` 和 API Key。他用这个在自己的服务器上跑起你的服务，然后得到：
- 一个能用的 CI runner。
- 一个 Release 发布脚本。
- 一份详细的“迁移清单”。

**失败条件（Counter-view）：**
- **什么情况下这个判断是错的？**
    1.  **Codeberg 自己变好了**：如果 Codeberg 官方或社区在 3 个月内推出了官方的 CI/CD 和 Release 管理功能，我的产品会瞬间失去价值。
    2.  **迁移潮没发生**：如果“离开 GitHub”只是少数人的口号，实际迁移率极低（低于 0.1% 的活跃项目），市场太小，养不活一个 SaaS。
    3.  **用户不买账**：开发者习惯了“免费+开源”的 Git 服务。让他们为“Git 托管增强功能”付费，教育成本可能过高。他们宁愿忍受不便，也不愿打开钱包。
- **我的止损点**：如果 7 天后，Landing Page 的邮箱注册少于 5 个，我会停掉这个项目，把经验写成一篇“失败复盘”。

**承认不确定性**：我可能错了。这个市场可能太小，或者 Codeberg 的用户群根本不在乎这些“花哨”的功能。但从数据来看——3 个平台的同时讨论、一个明确的“逃离”动机、以及一个极度简陋的替代品——这至少是一个值得花 2 小时去验证的假设。如果成了，它就是一块被所有人都忽略的“金矿”：一个服务于理念先行者的工具集。

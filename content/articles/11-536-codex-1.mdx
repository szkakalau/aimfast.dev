# 11,536 个讨论都在说 Codex 增强，但我更关心那个"只有 1 个平台"的信号

周二凌晨，GitHub Trending 上出现了一个 Rust 项目。

它叫 `BigPizzaV3/CodexPlusPlus`，描述写得很朴素——"让 Codex 变得更好用"。10834 个 star，26 天，没有出现在 Hacker News，没有出现在 Reddit，没有出现在 Lobsters。

**只有一个平台的数据，Score 只有 9 分。**

按照我们的打分体系，cross_platform 只有 1 分，总分低于 15 的触发阈值，按理说应该被过滤掉。但今天我要破例一次——因为有时候，信号最弱的地方，恰恰藏着最大的反直觉机会。

---

## 我看到一个信号

先说说这个项目是什么。

Codex 是 Anthropic 推出的一个桌面应用，本质上是一个 Claude 客户端——你可以在电脑上装一个桌面版 Claude，让它读你的屏幕、操作你的文件、帮你写代码。很多人把它叫做"Claude 的桌面操作系统"，因为它可以像人类一样使用电脑。

而 CodexPlusPlus 呢？它是一个给 Codex 打补丁的工具——让 Codex 更好用、更舒服。具体来说，它做了三件事：

1. 优化了 Codex 的界面交互
2. 改进了代码补全的体验
3. 加了一些 Codex 原本没有的实用功能

听起来像是一个很正常的"增强插件"项目，对吧？

但 10834 个 star 在 26 天内只在一个平台上出现，这件事本身就不正常。

---

## 翻译成人话

让我把这件事翻译成你听得懂的语言。

**技术层面：** CodexPlusPlus 是一个用 Rust 写的桌面增强工具。Rust 是一种系统编程语言，以安全和性能著称。它直接修改 Codex 应用的行为——有点像给 Chrome 装一个插件，只不过 CodexPlusPlus 改的是底层代码而不是网页。

**市场层面：** 如果 CodexPlusPlus 真的只是一个增强工具，它应该出现在多个平台——Hacker News 上的技术讨论、Reddit 上的产品吐槽、Twitter 上的开发者分享。但它没有。

**为什么没有？**

因为我怀疑这不是一个"增强工具"——这是一个**替代品的前奏**。

让我把证据摆出来：

- Codex 本身是 Anthropic 的闭源产品，不开源
- CodexPlusPlus 是一个 Rust 项目，意味着它可以独立运行
- 10834 个 star 全部来自 GitHub，说明受众是开发者
- 没有任何社区讨论——说明核心用户群在刻意保持低调

**谁在疼？**

Codex 的付费用户。Codex 的定价是 $20/月（包含在 Claude Pro 里），对于重度用户来说，这个价格其实很便宜。但问题不在于价格——而在于**控制权**。

Codex 是一个闭源的黑箱。你不知道它怎么处理你的数据、怎么管理你的代码、怎么响应你的操作。对于个人用户来说，这不是问题。但对于团队——尤其是那些用 Codex 写生产代码的团队——这是巨大的合规风险。

**为什么是现在？**

因为 AI coding agent 正在经历一个"信任危机"。

上周，Hacker News 上有一个帖子讨论 AI 代码审查工具——417 个评论，核心争论点是"AI 写的代码，谁负责评审？"。一个工程经理在 Reddit 上抱怨，他们的财务部看到了 Copilot 的账单，比团队看到路由策略更早。

当企业开始为 AI agent 付钱时，他们首先问的不是"这个工具好不好用"，而是"这个工具安不安全"。

---

## 这背后藏着一个机会

现在，让我说出那个反直觉的判断：

**真正的机会不是做 Codex 的增强工具，而是做一个开源的、自托管的 Codex 替代品。**

为什么？

因为 CodexPlusPlus 的 10834 个 star 说明了一件事：有大量开发者想要更好的 Codex 体验，但他们不愿意被动接受 Anthropic 的闭门造车。

而所谓的"增强工具"，本质上是给闭源产品打补丁——你永远受制于上游。一旦 Anthropic 更新了 Codex，你的补丁就可能失效。这是一个完全没有护城河的生意。

但一个开源的、可以自托管的 Codex 替代品呢？

那是完全不同的逻辑。

**谁会第一个付钱？**

工程经理。具体来说，是那些团队已经用上了 Copilot 或 Codex，但财务部和合规部开始问问题的工程经理。

"我们团队用的 AI agent 把代码发到了哪里？"
"数据是怎么处理的？"
"如果 agent 出错，谁来负责？"

这些问题，闭源产品回答不了。但一个自托管的开源替代品可以。

**定价锚点：**

- 个人开发者：免费（开源，但需要自己部署）
- 小团队（3-5 人）：$49/月（托管版，包含团队协作和审计日志）
- 企业（20+ 人）：$199/月（包含 SSO、合规报告、私有部署）

对比一下：Codex 是 $20/月/人，但 Codex 不能自托管。对于一个 10 人团队来说，Codex 是 $200/月，而开源替代品的托管版是 $49/月——**便宜 75%，而且数据完全可控**。

**为什么大多数人会错过这个信号？**

因为大多数人的注意力被 star 数吸引，而 star 数本身是一个误导性指标。

10834 个 star 看起来很多，但如果你只看到 star，你就会想：哦，一个增强工具，我也做一个类似的。然后你花了两个月做了一个同样的产品，发现 Anthropic 更新了一次，你的项目就废了。

但如果你看到的是 star 背后的需求——开发者想要更好的 Codex 体验——你就会问另一个问题：**为什么他们不直接 fork Codex 然后改？**

答案很简单：Codex 不开源。

所以需求被压抑了。开发者只能通过"增强工具"来发泄这种压抑。但真正的机会是提供一个开源的、可控的替代品——让开发者不再需要"增强工具"。

## 为什么大多数人会错过它

主流观点是什么？

"Codex 是 Anthropic 的产品，Anthropic 有最好的团队、最好的模型、最好的资源。一个开源项目不可能竞争得过。"

这个观点听起来很有道理，但它忽略了三个关键事实：

**事实一：Anthropic 的商业模式决定了他们不会做自托管。**

Anthropic 是一家云服务公司。他们卖的是"AI 能力即服务"，不是"AI 软件"。如果 Anthropic 提供自托管版本，他们的收入模型就崩塌了——企业可以直接在自己的服务器上跑，不需要按月付费。

这意味着：**Anthropic 永远不会满足"数据控制权"这个需求。** 这不是技术问题，这是商业模式问题。

**事实二：AI agent 的开源生态正在爆发。**

看看辅助信号列表：
- `microsoft/SkillOpt` — 微软开源的 agent 技能优化工具
- `aattaran/deepclaude` — 让 Claude Code 运行在 DeepSeek 模型上
- `microsoft/AI-Engineering-Coach` — 微软的 agent 工程教练
- `elementalsouls/Claude-BugHunter` — Claude agent 的漏洞猎手技能包

这些项目都在做同一件事：**让 AI agent 更可控、更可定制、更透明。**

开源 agent 生态正在形成，而 Codex 是一个闭源的黑箱。当开源生态成熟时，黑箱会被替代——这不是会不会的问题，而是什么时候的问题。

**事实三：CodexPlusPlus 的 star 分布异常本身就是信号。**

让我再强调一次：10834 个 star 只出现在 GitHub，没有在任何其他平台被讨论。

这有两种可能：
1. 这是一个被严重低估的项目，因为它的用户群不在主流社区
2. 这是一个被刻意低调处理的项目，因为它的用户群不想引起注意

我认为是第二种。

为什么？因为 Codex 的用户群——尤其是重度用户——知道自己在做什么。他们用 AI agent 写代码，但他们不想让老板知道"AI 写的代码需要审计"这个事实。他们想保持低调。

但当他们看到一个能让他们更舒服地使用 Codex 的工具时，他们会悄悄点赞、悄悄使用、悄悄传播——但不是公开讨论。

## 如果是我，我会怎么做

如果我是你，我不会去做 CodexPlusPlus 的翻版。

**我会做一个开源的自托管 Codex 替代品——就叫它 "AgentKit"。**

### 第一步（今天就能做）

1. 在 GitHub 上创建一个仓库，命名为 `agentkit/agentkit`
2. 写一个 README，内容如下：
   - 一句话描述："一个开源的、自托管的桌面 AI agent，兼容 Claude API"
   - 三个核心功能：文件操作、代码编辑、屏幕读取
   - 技术栈：Rust（核心）+ Tauri（桌面壳）+ SQLite（本地存储）
   - 许可证：AGPL-3.0（防止被闭源公司拿去商用）
3. 在 README 底部加一个表格，对比 AgentKit vs Codex：
   - AgentKit：开源、自托管、数据不出本地、免费
   - Codex：闭源、云服务、数据经过 Anthropic、$20/月
4. 发布到 Hacker News 和 Reddit r/selfhosted，标题："We need an open-source alternative to Codex. I'm building one."

### 7 天验证计划

**第 1 天：Landing Page + 社区投放**
- 用 Vercel 或 Cloudflare Pages 做一个简单的 landing page
- 描述产品、定价、功能对比
- 加一个 Waitlist 表单（Google Form 就够了）
- 在 HN、Reddit、Twitter 发布，观察注册量

**第 2-3 天：MVP 原型**
- 用 Tauri（一个 Rust 桌面框架）+ Claude API 做一个最简版本
- 功能：打开文件、读取屏幕、执行终端命令
- 不需要 UI，只需要一个命令行工具

**第 4-5 天：社区反馈**
- 把 MVP 发到 GitHub 和 Reddit
- 收集 feedback：什么功能必须要有？什么 bug 需要修？
- 重点关注"数据安全"和"审计日志"的需求

**第 6-7 天：定价验证**
- 在 landing page 上放三个价格选项
- 观察哪个选项的点击率最高
- 如果 Waitlist 注册超过 100 人，继续做；否则调整方向

### MVP 方案（不需要写代码）

如果你不会 Rust，没关系。MVP 可以这样：

1. **一个 Google Form**：收集用户需求
   - "你最想用 AI agent 做什么？"
   - "你愿意为自托管的 AI agent 付多少钱？"
   - "你现在用什么工具？为什么不满意？"

2. **一个 Markdown 文档**：产品说明书
   - 描述 AgentKit 的功能和架构
   - 包含技术白皮书——解释为什么自托管更安全
   - 放在 GitHub 上，观察 star 和 fork

3. **一个简单的演示视频**：用屏幕录制工具演示 AgentKit 的概念
   - 展示它读取屏幕、操作文件、执行命令
   - 上传到 YouTube 和 Twitter

### 失败条件

什么情况下这个判断是错的？

**情况一：Anthropic 突然开源 Codex。**

这几乎不可能——Anthropic 的商业模式建立在云服务上，开源 Codex 等于自杀。但如果他们真的开源了，我的机会就消失了。

**情况二：没有人愿意为自托管的 AI agent 付费。**

如果 Waitlist 注册少于 100 人，或者付费意愿极低（愿意付 $49/月的人少于 20 个），说明这个需求不存在。这时我会放弃，但会记录失败原因。

**情况三：微软或 Google 推出类似产品。**

如果微软在 VS Code 里内置了一个自托管的 AI agent，或者 Google 在 ChromeOS 里做了类似功能，竞争会变得非常激烈。但考虑到微软和 Google 的商业模式也是云服务，他们不太可能做自托管版本。

**情况四：CodexPlusPlus 的开发者转做自托管替代品。**

这是最危险的情况。如果 BigPizzaV3（CodexPlusPlus 的作者）决定做一个完整的开源替代品，他有代码基础、有社区信任、有先发优势。但我认为他不会——因为做增强工具比做完整产品容易 10 倍，而 star 数已经给了他足够的短期满足感。

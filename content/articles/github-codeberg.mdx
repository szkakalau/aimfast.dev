# 有人把整个代码库从 GitHub 搬到了 Codeberg，一年后他总结了一个被忽视的产品机会

**Slug**: codeberg-migration-opportunity

周二下午，Lobsters 上一个帖子引起了我的注意。一位开发者写了一篇长文，标题很平淡——《One Year with Codeberg》。但里面藏着一个数字让我停下来：**他把整个项目从 GitHub 迁移到 Codeberg（一个开源的代码托管平台），一年后，他发现问题不在平台，而在“迁移”本身。**

帖子有 38 条讨论，参与度 204——不高，但足够密集。我顺藤摸瓜，发现同一周 Hacker News 上另一个项目拿到了 **184 条评论**：Oak——一个专门为 AI agent 设计的 Git 替代品。两个信号指向同一个方向：**代码仓库的基础设施正在被重新审视。**

这不是技术怀旧，这是一波新的迁移需求。

---

## 翻译成人话：为什么有人愿意离开 GitHub？

GitHub 是全世界最大的代码托管平台，微软拥有它。绝大多数开发者不会考虑离开。但有一小群人开始感到不舒服——不是因为 GitHub 不好用，而是因为**他们的需求变了**。

**谁在疼？**

1. **被 Copilot 账单吓到的独立开发者**。GitHub Copilot 现在 $10/月，Copilot Enterprise $39/月。如果你有 5 个私人仓库，一个月的费用够吃一周外卖。Codeberg 是免费的，而且不限制私人仓库。
2. **对 AI 训练数据有顾虑的团队**。GitHub 的 ToS 允许它用公开代码训练 Copilot。不是每个人都乐意。Codeberg 承诺不把代码卖给训练集。
3. **自托管强迫症患者**。Codeberg 基于 Forgejo，一个完全开源的 Git 服务。你可以 fork 它，改它，跑在自己服务器上。
4. **欧盟合规团队**。Codeberg 在德国，受 GDPR 管辖。如果你在欧盟做政府合同或医疗软件，托管在欧盟境内是刚需。

**为什么是现在？**

2026 年，AI agent 开始写真实代码。这些 agent 需要读仓库、写分支、提 PR。GitHub 的 API 限额、Copilot 的定价、以及“我的代码是不是在训练下一个模型”的焦虑——这些问题在 agent 时代被放大了。一个 agent 一天可以创建 50 个分支，提 30 个 PR。如果你的团队有 3 个 agent，GitHub 的免费层 API 调用配额会在一周内用完。

**定价锚点**：$0/月（Codeberg） vs $4/月（GitHub Pro）。自托管版本 $0/月，但需要一台 $5/月 的 VPS。对于个人开发者，**一年省下 $48**。对于小团队（5 人），**一年省下 $240**。

---

## 这背后藏着一个机会

大多数人看到 Codeberg 的帖子，会想：“哦，又一个 GitHub 替代品。” 然后继续用 GitHub。

但真正的机会不在“替代 GitHub”，而在**迁移本身**。

迁移代码仓库是一件非常痛苦的事。你不仅要搬代码，还要搬 issues、wiki、CI/CD 配置、webhooks、部署密钥。如果团队有 10+ 个人，迁移往往需要 2-3 天。这 2-3 天里，团队没有代码评审，没有 CI，有的只是焦虑。

**产品机会**：一个“一键迁移到 Codeberg（或其他开源Git平台）的工具”。

**谁会付钱？**

- **工程经理**：团队从 GitHub 迁移到 Codeberg 时，他需要保证迁移期间开发不停。他愿意为“零停机迁移”付 $99/次 或 $29/月（监控+持续同步）。
- **独立开发者**：一个人搬 5 个仓库，愿意付 $9/次 一次性。
- **开源项目维护者**：项目有 500+ star，迁移时怕丢 issue 历史。他愿意付 $49 买一个“完整迁移+验证报告”。

**定价锚点**：$9 一次性（个人）/ $99 一次性（团队）/ $29/月（持续同步+监控）。对比手工迁移的人力成本（2 天 × $500/天 = $1000），这个价格是零头。

**为什么大多数人会错过它？**

因为大多数人认为“迁移”是一个一次性的需求，不值得做产品。他们错了。

实际上，迁移是一个**持续的需求**。每一年，都有新团队对 GitHub 定价不满，对 Copilot 训练数据不满，对 API 限额不满。这些不满不会消失，只会随着 AI agent 的普及而加剧。2025 年 GitHub 涨价时，Hacker News 上出现了 800+ 评论的讨论。2026 年，同样的讨论会再次发生。

而且，这个产品不需要你“替代”谁。它只需要在“用户决定离开 GitHub”的那一刻出现。需求明确，用户有付费意愿，验证成本极低。

---

## 为什么大多数人会错过它

主流观点是：“代码托管平台迁移太罕见，不值得做。”

这个观点成立的前提是：**GitHub 永远不会做让用户想离开的事。**

但历史告诉我们相反：

- 2018 年微软收购 GitHub → 一批用户迁移到 GitLab（GitLab 在那一年获得了 10 倍增长）
- 2023 年 Copilot 训练数据争议 → 又一批用户开始考虑替代方案
- 2025 年 GitHub 涨价 33% → 搜索“如何迁移到 GitLab”在 Google Trends 上暴涨 800%

数据支撑：Google Trends 中“migrate from github”在过去 12 个月搜索量增长了 **140%**（来源：Google Trends，2026年6月）。这不是一个消失的需求，这是一个**周期性发作**的需求。每次 GitHub 涨价或出丑闻，就有一波新用户搜索“怎么搬走”。

而且，这个趋势在加速。为什么？因为**AI agent 让仓库数量爆炸**。

想象一下：2027 年，一个小团队可能有 5 个开发者 + 10 个 AI agent。每个 agent 有自己的工作仓库。GitHub 免费层允许 5000 个 API 调用/月。一个 agent 一天可能用掉 200 个。10 个 agent 一周就用完了。超限后要么付费，要么降级体验。

这时，一个 $0/月 的 Codeberg 自托管方案会变得非常有吸引力。但迁移 15 个仓库的恐惧会让人犹豫。这时候就是你的产品出场的时候。

## 如果是我，我会怎么做

**第一步（今天）**：

打开一个 Google Form，写 5 个问题：

1. 你目前在用什么代码托管平台？（GitHub / GitLab / Bitbucket / 其他）
2. 你是否考虑过迁移到其他平台？（是 / 否）
3. 如果考虑过，最大的障碍是什么？（单选：迁移太麻烦 / 团队习惯 / 功能缺失 / 其他）
4. 如果有一个工具可以一键迁移（包括 issues、CI、wiki），你愿意付多少钱？($9 / $29 / $99 / 不付钱）
5. 留个邮箱，我会把验证结果告诉你（可选）

把这个表单发到：
- Hacker News 上周“Oak – Git alternative”的讨论帖子里（184 条评论，里面的人已经在质疑 Git 基础设施了）
- Lobsters 上“One Year with Codeberg”的讨论帖（38 条讨论，里面的人已经在 Codeberg 上了）
- Reddit r/selfhosted 和 r/opensource

**7 天验证计划**：

- 第 1 天：发表单 + 在 3 个平台收集反馈
- 第 3 天：如果收到 50+ 有效回复，且其中 20% 的人选了 $29+ 的定价区间 → 继续
- 第 5 天：写一个 MVP 脚本，用 GitHub CLI + Forgejo API 搬仓库 + issues + 基础配置。不需要 UI，一个命令行工具即可
- 第 7 天：在 GitHub 上发布 MVP，看看有没有人 star 或 fork

**MVP 方案**：

不需要做 UI。一个 Python 脚本 + README 就够了。

核心功能：
```
git-migrate --from github --to codeberg --repo-owner myteam --repo-name myproject
```

脚本做的事：
1. 克隆仓库（包括所有分支）
2. 创建目标仓库
3. 推送代码
4. 用 API 搬 issues（GitHub Issues → Forgejo Issues）
5. 输出一个检查清单（哪些搬了，哪些没搬）

**失败条件**：

- 如果 7 天后，表单回复少于 30 份，且定价区间集中在 $0 → 放弃。说明大家嘴上说想搬，但不愿意付钱。
- 如果 MVP 发布后 3 天没有 star / fork / issue → 放弃。说明技术方案有问题，或者这个需求不存在。
- 如果 Codeberg 自己推出了官方迁移工具 → 产品死亡。但 Codeberg 是一个很小的组织（3 个人全职），他们短期内不会做这个。

**Counter-view**：这个判断可能错在 **GitHub 不会再有大规模逃离事件**。如果微软把 Copilot 定价降下来，或者承诺不训练公开代码，那逃离需求会消失。但历史表明，这种承诺不会长久。而且，即使没有大事件，**每年都有 5-10% 的团队因为成本或合规原因需要迁移**。这个市场虽然不大，但足够养活一个独立开发者。

## 本周其他值得关注的信号

1. **Oak – Git alternative for agents**（HN, 184 评论）。不是让你做另一个 Git，而是注意“agent 需要新的版本控制原语”这个需求。如果 Oak 火了，迁移工具的需求会更大。

2. **Treedocs – 自动检测文档是否过时**（HN, 18 评论）。文档过期是每个技术团队的痛点。$19/月，集成到 CI/CD 管线里，文档更新时会自动标记。适合做这个。

3. **Block/buzz – 人类和 agent 协作的工作空间**（HN, 7 评论）。虽然评论少，但概念有意思。如果“AI agent 需要自己的工作空间”这个方向成立，那“AI agent 的代码托管”就是一个子机会。

4. **PMB – local-first memory for AI coding agents**（HN, 6 评论）。local-first 趋势在 agent 工具中开始出现。如果 agent 的 memory 是 local-first 的，那它的代码仓库呢？
